第七章 链接

链接是将代码和数据片段组合成一个单一文件的过程，这个文件可以被load到内存并执行。

链接使得分离编译成为可能，将大型应用程序分成多个小的模块，当更改其中一个小模块时，只需要重新编译它，再链接，而不必去编译其他文件。

编译器驱动程序包括了预处理器、编译器、汇编器和链接器。它将ASCII码源文件翻译成可执行目标文件。

预处理器cpp将main.c翻译成一个ASCII码的中间文件main.i。

目标文件

  • 可重定位目标文件：包含二进制代码和数据，可在编译时与其他可重定位目标文件合并，形成一个可执行目标文件。
  • 可执行目标文件：包含二进制代码和数据，可被加载到内存执行。
  • 共享目标文件：特殊可重定位目标文件，可在加载时或运行时被动态加载到内存并链接。

各个系统目标文件格式不同，x86-64Linux和Unix使用的格式为ELF。

  • .text：代码
  • .rodata：只读数据
  • .data：初始化的全局和静态变量。局部变量在运行时被保存在栈中，并不在.data和.bss中
  • .bss：未初始化的全局和静态变量、被初始化为0的全局和静态变量。这在目标文件中不占用空间，在运行时才会给变量分配内存，初始值为0
  • .symtab：符号表

每个可重定位目标文件包含一个符号表，其中符号表包含了3中不同符号：

  • 被该文件定义和引用的全局符号：该文件中定义的全局函数和非静态全局变量
  • 引用其他目标文件的全局符号：其他文件中定义的全局函数和非静态全局变量
  • 该文件定义和引用的局部符号：static修饰的变量和函数，这些可被该目标文件使用，但不能在其他目标文件中被引用。

链接静态库

将相关的目标文件打包成一个单独的文件就形成了静态库，当链接器链接静态库时，它只复制应用程序引用的静态库中的模块，未引用的不会复制，这减少了可执行文件的占用空间。Linux静态库以.a结尾标识。

重定位

链接器完成符号解析后就把每个模块中每个符号引用与它的定义相关联。

  • 重定位节和符号定义。将类型相同的节，如每个模块的.data节合并成为一个新的.data节。将运行时内存地址设置给新的节和每个符号。这样每条指令和全局变量就有唯一的运行时内存地址了。
  • 重定位节中的符号引用。修改代码和数据节中对每个符号的引用，使之指向正确的运行时地址。

重定位完后，形成可执行目标文件，它连续的片被映射到连续的内存段。

加载器通过目标文件头部表中的信息将进程虚拟地址空间中的页映射到可执行文件中页大小的片，新的代码段和数据段被初始化为可执行文件的相应段，然后加载器跳转到_start地址，它最终调用main函数，除了头部表的一些信息，加载过程中没有任何从磁盘到内存的数据复制。知道CPU引用一个被映射的虚拟页时才会复制相应页面到内存。

动态链接共享库

静态库的一些代码会被复制到每个引用它的运行的进程中，而动态库不同，它在运行或加载时，可以通过动态链接器加载到进程的任意内存地址，在Linux中以.so结尾的文件表示。所有引用一个.so动态库的可执行目标文件共享这个.so文件中的代码和数据。静态库则是每次链接都需要复制代码和数据。共享库的.text节可以被多个运行的进程共享

第八章 异常处理流

一段连续处理器指令的地址序列成为控制流（控制转移序列），就是一个接一个执行指令。操作系统使用异常控制流ECF（控制流突变，也就是指令跳转执行）对一些系统状态的变化做响应。应用程序使用陷阱（trap）或者系统调用的ECF形式，向操作系统请求服务。

软件异常允许程序进行非本地跳转（违反通常的栈规则）来响应错误，属于应用层ECF，如C++的try catch。

异常

控制流的突变，响应处理器状态的变化（事件）。简单来说就是跳到别的地址去执行指令，返回来是可选的。如图：

当处理器检测到事件时，就会进行一个间接过程调用，通过异常表和事件对应的编号，来执行这类事件的异常处理程序。处理完后，有以下选择：

  • 返回被中断的指令，并重新执行它
  • 返回被中断的指令的下一条指令
  • 终止被中断的进程

异常处理

每种类型异常分配一个异常号。操作系统启动时初始化异常表。

  • 在执行间接过程调用时，转入异常处理程序之前，处理器在栈中压入返回地址和处理器状态，如果控制是从用户态转移到内核态，就会被压入内核栈。
  • 异常处理程序运行在内核态。
  • 执行完毕后，执行中断返回指令，该指令恢复处理器状态，将控制权交给被中断的进程。

异常类别

  1. 中断：异步发生的硬件中断，处理外设I/O信号的结果，执行中断处理程序。
  2. 陷阱和系统调用：同步发生，是执行当前指令所产生的结果。应用程序执行syscall n指令请求对应的内核服务。（执行该指令会导致一个到异常处理程序的陷阱，该处理程序解析参数，调用适当内核服务程序）系统调用运行在内核模式。
  3. 终止：同步发生，调用abort终止进程。

  1. 故障：出现错误，可能被故障处理程序修复。若能修复就返回去重新执行引起故障的指令；否则调用abort终止进程。

典型例子：缺页异常（page falt）

指令引用了一个物理页面不在内存中的虚拟地址，缺页处理程序从磁盘加载对应页面到内存，然后重新执行引起缺页错误的指令。

Linux系统调用

每个系统调用有唯一系统调用号，这对应到内核中的跳转表的偏移量。

标准C库提供的包装函数如：read、write等。它们打包参数，并调用syscall n指令陷入内核，去执行系统调用，然后将返回值传递回来。

所有Linux系统调用的参数通过寄存器传递而不是栈。

进程

进程上下文由程序运行所需的状态组成，包括：

  • 代码段、数据段、堆、栈。
  • 寄存器内容、程序计数器。
  • 环境变量。
  • 打开文件描述符。
  • 等等。

进程提供了两个抽象：

  • 独立的逻辑控制流
  • 私有的地址空间

用户模式和内核模式

CPU使用控制寄存器的一个模式位来区分。用户程序必须使用系统调用接口间接访问内核代码和数据。进程从用户模式变为内核模式的唯一方法是通过中断、故障或陷入系统调用这样的异常。在异常处理程序中，CPU将用户模式变为内核模式。

proc文件系统

存放内核数据结构的相关信息如：进程使用的内存段(/proc//maps)。

上下文切换

内核使用上下文切换的异常控制流实现多任务。内核为每个进程维护上下文，就是重新启动被抢占进程所需的状态。包括程序计数器、寄存器、用户栈、内核栈和各种内核数据结构，比如描述地址空间的页表、进程表、打开文件表等。

切换步骤：

  1. 保存当前进程上下文。
  2. 恢复之前被抢占进程的上下文。
  3. 将控制传递给恢复进程。

第九章 虚拟内存

虚拟内存为进程提供了格式一致的私有地址空间。

  1. 将主存看成是磁盘上地址空间的高速缓存，只保存活动区域，高效。
  2. 为每个进程提供一致的私有地址空间，简化内存管理。
  3. 保护每个进程地址空间。

malloc管理虚拟存储分配。

物理和虚拟寻址

主存是连续的字节数组，每个字节有自己的地址。
现代CPU采用虚拟寻址：CPU生成虚拟地址送给MMU，MMU根据VA查询主存中的页表生成PA，再根据物理地址从主存中取数据，如图：

虚拟内存作为缓存工具

VM被分成大小固定的虚拟页（VP），PM被分成大小固定的物理页（PP页帧）。通常两者大小相同。

虚拟页面集合的种类：

  1. 未分配的页：没有任何数据与之关联。
  2. 缓存的页：已缓存在物理内存中。
  3. 未缓存的：未缓存在物理内存中的已分配的页，引发page fault。

下图为大小为P字节的页：

DRAM缓存

DRAM缓存不命中代价高，因为使用磁盘服务，从一个磁盘扇区读取第一个字节的速度比读取连续的字节要慢太多。因此虚拟页要足够大来提高DRAM命中率，减少对磁盘的访问，而且DRAM使用全相联即：任何虚拟页可以放置在任何物理页中。
写入物理页的数据并不直接写入磁盘，而是先存在物理页，之后某个时间一起写回到磁盘，这减少了对磁盘的访问，加快了速度。

页表

  • 确定虚拟页是否缓存在DRAM中Valid bit。
  • 确定虚拟页存放在哪个物理页PPN。
  • 若未命中，还需确定虚拟页在磁盘的位置。
    这就需要页表了。
    每个进程拥有自己的页表，它将虚拟页映射到物理页，每次MMU翻译地址时都会读取页表。操作系统维护页表，并在磁盘和DRAM间来回复制页。

页命中

访问VP2虚拟内存的数据时，页表中VP2的Valid bit = 1，表明该虚拟页被缓存在DRAM中，则可根据VP2的PTE的PPN找到对应物理页地址从而访问数据。

缺页（Page Faults）

如果访问VP3中的数据，VP3的PTE的PPN != null表明该虚拟页已被分配，Valid bit = 0表明该虚拟页未被缓存到物理内存，这就引发缺页异常（DRAM未命中），内核调用缺页异常处理程序，它会选择一个牺牲页VP4，如果该牺牲页的脏位是1，表明它被修改过，那么就要先将该页写回到磁盘，然后将VP4的PTE的Valid bit置为0，再从磁盘拷贝VP3的内容到物理页PP3并修改VP3的PTE内容。然后重新执行引发缺页异常的指令。

在磁盘和内存传送页的活动称为交换或页面调度，通常使用按需页面调度策略：一直等到，直到未命中发生时才换入页面。

分配页面

例如调用malloc，内核在磁盘上创建VP5虚拟页面，并更新PTE5。
注意：这并不将磁盘页面复制到物理内存中。

虚拟内存作为内存管理的工具

  • 简化内存管理
  • 保护内存
    多个虚拟页可以映射到一个共享的物理页面。
  1. 简化链接。每个进程拥有一致格式的独立地址空间。
  2. 简化加载。包括可执行文件和共享对象文件，只需通过分配虚拟页，并更改PTE即可，并不需要复制磁盘数据到内存中（按需页面调度）。
  3. 简化共享。例如，每个进程共享操作系统的代码。这通过将不同进程的虚拟页面映射到相同的物理页面实现（修改PTE），而在内存之只需要一份共享代码即可。
  4. 简化内存分配。进程要求额外堆空间时，只需分配虚拟页面，并映射到物理页面即可。

虚拟内存作为内存保护的工具

在页表的PTE中添加几个访问权限位（超级、读、写）。

地址翻译

CPU中的页表基址寄存器，指向当前页表。

页命中，CPU的执行步骤：

  1. CPU生成虚拟地址，交给MMU。
  2. MMU生成PTE地址，发给主存。
  3. 主存返回PTE内容给MMU。
  4. MMU根据PTE的PPN和VPO生成物理地址发给主存。
  5. 主存返回请求字给CPU。

缺页，CPU的执行步骤：

  1. 同上1-3.
  2. MMU发现PTE的Valid bit = 0，MMU触发缺页异常，将控制传递给操作系统内核的缺页异常处理程序。
  3. 处理程序找出牺牲页，如果脏位为1，还要将牺牲页写回磁盘。
  4. 处理程序调入新页面，更新PTE。
  5. 返回到原进程，再次执行导致缺页的指令。CPU再次将虚拟地址发给MMU，这次由于该页被缓存，所以会正常命中。

结合高速缓存和虚拟内存

大多数高速缓存采用物理寻址。
